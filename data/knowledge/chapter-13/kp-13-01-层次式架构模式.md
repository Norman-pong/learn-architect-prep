## 概述

层次式架构（Layered Architecture）是软件体系结构设计中最常用的一种架构形式，其核心思想是将系统组织为一个层次结构，每一层为上层服务、并作为下层客户[^1]。它通过关注点分离（separation of concerns）将应用划分为表现层、中间层、数据访问层和数据层等，每一层只负责本层逻辑，从而降低复杂度并提高可维护性[^2]。层次式架构也被称为 N 层架构模式（n-tier architecture pattern），是 Java EE / .NET 等企业应用的首选初始架构[^3]。

## 核心知识点

### 1 层次式架构基本结构与特点

层次式架构本身不规定必须分多少层，但典型应用会分为四层[^4]：

| 层次 | 职责 | 与其他层关系 |
|------|------|--------------|
| 表现层 / 展示层 | 负责用户交互与界面呈现 | 调用中间层 |
| 中间层 / 业务层 | 封装业务规则与流程 | 调用访问层，被表现层调用 |
| 访问层 / 持久层 | 负责数据持久化与外部数据访问 | 调用数据层，被中间层调用 |
| 数据层 | 实际数据库、文件或外部系统 | 被访问层调用 |

层次式架构的两个关键设计注意事项：

- **污水池反模式（architecture sinkhole anti-pattern）**：请求只是穿过多个层而没有在层内做实际业务逻辑。若此类请求超过 20%，应考虑让某些层变成开放的，否则可维持封闭层[^5]。
- **避免过度分层**：分层可能让应用变得庞大，带来分布复杂、健壮性下降、可靠性与性能不足、代码膨胀等问题[^6]。

### 2 表现层设计模式：MVC / MVP / MVVM

教材在本章重点介绍了三种表现层设计模式，它们均用于实现视图与业务逻辑的分离[^7]。

#### 2.1 MVC 模式

MVC 将输入、处理、输出流程按**视图（View）— 控制（Controller）— 模型（Model）**分离[^8]：

- **Controller（控制器）**：接收用户输入并调用模型和视图完成需求；解释视图输入并转换为对模型的方法调用；处理模型事件并调用适当视图给出反馈[^9]。
- **Model（模型）**：表示业务数据和业务逻辑，是应用主体；一个模型可为多个视图提供数据，提高可重用性[^10]。
- **View（视图）**：用户看到并与之交互的界面；显示模型数据、接收用户输入，但不进行实际业务处理；视图可查询模型状态，但不能直接修改模型[^11]。

MVC 的优点包括：允许多种用户界面扩展、易于维护、功能强大的用户界面、业务处理与显示分离，提升了可扩展性与灵活性[^12]。

#### 2.2 MVP 模式

MVP（Model-View-Presenter）由 MVC 演变而来，核心变化是 **View 与 Model 不直接通信**，所有交互都通过 Presenter 进行[^13]：

- Presenter 依赖的是抽象化的 View（即 IView 接口），使 UI 处理逻辑易于单元测试[^14]。
- 优点：模型与视图完全分离；一个 Presenter 可用于多个视图；逻辑可脱离用户接口进行测试；在 Android 开发中应用较多[^15]。

#### 2.3 MVVM 模式

MVVM（Model-View-ViewModel）为解决 MVP 中 UI 种类增多导致接口不断增加的问题而提出[^16]：

- **ViewModel** 是核心，通过 **Data Binding** 实现 View 与 Model 的双向绑定[^17]。
- View 与 Model 不能直接通信，通信只能通过 ViewModel 实现；ViewModel 通常实现观察者，监听数据变化并通知视图自动更新，也能监听视图变化再通知数据改动[^18]。
- 适用场景：数据驱动、数据操作特别频繁的场景。缺点：双向绑定导致问题定位困难，可能来自数据问题，也可能来自业务逻辑对视图属性的修改[^19]。

### 3 数据访问层五种模式

教材第 13.4 节总结了数据访问层的 5 种数据访问模式[^20]：

| 模式 | 要点 |
|------|------|
| 在线访问 | 最常用，占用数据库连接，持续与后台数据源交互 |
| DAO（Data Access Object） | 将底层数据访问操作与高层业务逻辑分离，典型组件含 DAO 工厂、DAO 接口、具体实现类、数据传输对象 |
| DTO（Data Transfer Object） | 跨进程/网络边界传输数据的容器，不含业务逻辑，可用内置集合或自定义类实现 |
| 离线数据模式 | 数据从数据源获取后按预定义结构存放，独立于连接与事务，可与 XML 集成 |
| 对象/关系映射（O/R Mapping） | 将对象数据与关系数据库记录相互转换，用于持久层设计 |

## 考点与易错点

1. **MVC 与 MVP 的关键区别**：在 MVC 中，View 可以直接从 Model 读取数据；而在 MVP 中，View 与 Model 不直接通信，所有交互必须通过 Presenter。考试中常以“能不能直接通信”作为判断标准[^21]。
2. **MVVM 的双向绑定**：ViewModel 通过 Data Binding 实现 View 与 Model 的双向绑定，适用于数据驱动场景；但双向绑定也会增加问题定位难度[^22]。
3. **层次式架构的反模式**：污水池反模式指请求逐层穿过但几乎不做业务逻辑，若比例超过 20%，应考虑开放层。注意“20%”这个数字本身不是硬性阈值，而是判断参考[^23]。
4. **数据访问模式容易混淆**：DAO 是访问抽象与逻辑分离，DTO 是跨边界数据容器，O/R Mapping 是对象与关系记录的转换，三者定位不同[^24]。

## 真题示例

**【Q52】MVC 架构中，负责接收用户输入并更新模型的是（  ）。**

A. Model  
B. View  
C. Controller  
D. Router

**答案：C. Controller**

**解析**：MVC 模式中，Controller 接收用户输入，将其解释并调用 Model 完成业务处理；View 只负责展示，Model 只负责业务数据与逻辑，Router 不是 MVC 三要素之一。本题对应教材第 13.2.1 节 MVC 模式的内容[^25]。

## 记忆要点

> **口诀：层上分层，MVC 三兄弟；MVP 加中介，MVVM 靠绑定。**

- 分层：表现层 → 中间层 → 访问层 → 数据层，上层依赖下层，关注分离。
- MVC：Controller 接输入、Model 管数据逻辑、View 管展示；View 可查询 Model。
- MVP：View ↔ Presenter ↔ Model，View 不直接访问 Model，Presenter 依赖 IView，好测试。
- MVVM：View ↔ ViewModel ↔ Model，双向数据绑定，数据驱动场景首选。
