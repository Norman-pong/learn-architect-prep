## 第一部分：理论知识（严格按官方教材章节汇总）

### 第1章 绪论

**1.1 系统架构概述**
- **架构定义**：系统的骨架和灵魂，一组构件及构件之间的连接关系。
- **架构的演进**：从单体→分层→SOA→微服务→云原生。
- **架构分类**：
  - 按范围分：企业架构、系统架构、软件架构
  - 按层次分：业务架构、应用架构、数据架构、技术架构

**1.2 软件架构的分类与建模**
- **常见分类**：
  - 模块结构（Module）：子系统、模块、层的分解
  - 构件与连接件（C&C）：运行时交互
  - 分配式（Allocation）：软硬件映射关系
- **4+1视图模型（Philippe Kruchten, 1995）**：
  | 视图 | 描述 | 关注者 |
  |:---|:---|:---|
  | 逻辑视图 | 功能分解、类、接口 | 最终用户 |
  | 开发视图 | 包的层次结构、模块组织 | 开发者 |
  | 进程视图 | 并发、同步、通信 | 系统集成者 |
  | 物理视图 | 软硬件映射、部署 | 运维工程师 |
  | 场景（+1） | 用例驱动，串联四视图 | 所有干系人 |
- **架构建模语言（ADL）**：Wright、ArchStudio、xADL 等。

**1.3 系统架构设计师的角色与职责**
- 知识结构：技术广度（基础设施→中间件→业务）、架构决策、沟通协调。
- 能力要求：抽象思维、权衡取舍（Trade-off）、沟通表达。

### 第2章 计算机系统基础知识

**2.1 计算机硬件**
- CPU组成：运算器、控制器、缓存（L1/L2/L3）、寄存器。
- 存储体系：SRAM→DRAM→磁盘→磁带（速度与成本金字塔）。
- Flynn分类法：
  | 类别 | 指令流 | 数据流 | 示例 |
  |:---|:---:|:---:|:---|
  | SISD | 1 | 1 | 传统单核 |
  | SIMD | 1 | 多 | GPU、向量机 |
  | MISD | 多 | 1 | 容错系统（罕见） |
  | MIMD | 多 | 多 | 多核、集群 |

**2.2 操作系统**
- **进程管理**：三态（就绪/运行/等待）、五态（+新建/终止）、PCB。
- **PV操作与同步互斥**：信号量、生产者-消费者、读者-写者、哲学家进餐。
- **死锁**：4必要条件（互斥/保持并等待/非剥夺/循环等待）。银行家算法。
- **存储管理**：
  - 分区（固定/可变）、分页（页表/TLB/缺页）、分段、段页式。
  - 页面置换算法：FIFO（Belady异常）、LRU、LFU、CLOCK、最佳置换（OPT）。
- **文件系统**：目录结构、FCB、文件分配方法（连续/链接/索引）、空闲存储管理（位图/空闲链表）。
- **设备管理**：I/O控制方式（轮询/中断/DMA/通道）、SPOOLing、缓冲（单/双/循环）。

**2.3 计算机网络**
- **OSI七层模型** vs **TCP/IP四层模型**（常考对照表）。
- **协议要点**：
  - 应用层：HTTP 1.1/2.0/3.0（QUIC）、DNS、FTP、SMTP、DHCP
  - 传输层：TCP（三次握手/四次挥手/滑动窗口/拥塞控制）、UDP
  - 网络层：IP（IPv4/IPv6）、ICMP、ARP、NAT、CIDR
- **局域网与广域网**：以太网、VLAN、PPP。
- **网络安全基础**：对称加密（DES/AES）、非对称加密（RSA/ECC）、数字签名、CA/PKI、SSL/TLS、IPSec（AH/ESP）、防火墙（包过滤/状态检测/应用代理）、IDS/IPS。

**2.4 系统工程**
- 系统工程生命周期：概念→开发→生产→使用→保障→退役。
- 结构化分析与设计：DFD（数据流图）、ERD（实体关系图）、STD（状态转换图）。
- 系统性能评价：响应时间、吞吐量、利用率、Amdahl定律（加速比 = 1 / ( (1-F) + F/N )）。

### 第3章 信息系统基础知识

**3.1 信息系统概念与类型**
- 定义：输入→处理→输出的数据加工系统。
- 类型：T题的层次 → TPS → MIS → DSS → EIS。

**3.2 企业信息化**
- ERP（企业资源计划）：MRP→MRPⅡ→ERP→ERPⅡ。
- CRM（客户关系管理）：运营型/分析型/协作型。
- SCM（供应链管理）：推式/拉式、牛鞭效应。
- BI（商业智能）：OLAP、数据仓库、数据挖掘。

### 第4章 信息安全技术基础知识

**4.1 安全体系结构**
- OSI安全体系：5类安全服务（认证/访问控制/数据保密/数据完整/抗抵赖）。
- 安全等级保护（等保2.0）：GB/T 22239，5级（用户自主/系统审计/安全标记/结构化/访问验证）。

**4.2 密码学**
- 对称加密：DES（56位密钥）、3DES、AES（128/192/256）、SM4（国密）。
- 非对称加密：RSA（大整数分解）、ECC（椭圆曲线）、SM2（国密）。
- 哈希：MD5（已破解）、SHA-1（已弱化）、SHA-256/512、SM3。
- 数字签名：发送方私钥签名，接收方公钥验证，提供不可否认性。

**4.3 PKI与证书**
- CA（证书签发机构）、RA（注册机构）、证书库、CRL（证书撤销列表）。
- X.509证书格式。

### 第5章 软件工程基础知识

**5.1 软件生命周期模型**
| 模型 | 特点 | 适用 |
|:---|:---|:---|
| 瀑布 | 阶段清晰、不可逆 | 需求稳定项目 |
| 原型 | 快速构建、反复修改 | 需求模糊项目 |
| 增量 | 分块交付、逐步完善 | 大型系统 |
| 螺旋 | 风险驱动、每次循环含规划/风险分析/开发/评估 | 高风险项目 |
| 敏捷 | 迭代增量、自组织团队 | 需求频繁变化 |
| RUP | 用例驱动、架构中心、迭代增量 | 企业级开发 |
| DevOps | 开发运维一体化、CI/CD | 云原生/微服务 |

**5.2 需求工程**
- 需求层次：业务需求、用户需求、系统需求。
- 需求分类：功能需求、非功能需求（质量属性）、约束条件。
- 结构化分析方法：DFD+数据字典+结构化语言+判定表/树。
- 面向对象分析方法（OOA）：UML用例图、活动图。

**5.3 软件设计**
- 结构化设计：SD方法、结构化图表（SC图）。
- 面向对象设计（OOD）：职责分配、开闭原则、依赖倒置。
- 设计模式（23种GoF）：
  | 类别 | 模式 |
  |:---|:---|
  | 创建型（5） | 单例、工厂方法、抽象工厂、建造者、原型 |
  | 结构型（7） | 适配器、桥接、组合、装饰、外观、享元、代理 |
  | 行为型（11） | 模板方法、策略、观察者、迭代器、职责链、命令、备忘录、中介者、状态、访问者、解释器 |

**5.4 软件测试**
- 测试层次：单元→集成→系统→验收。
- 测试方法：黑盒（等价类/边界值/因果图/判定表）、白盒（语句覆盖/判定覆盖/条件覆盖/路径覆盖/MCDC）。
- McCabe圈复杂度：`V(G) = E - N + 2P` 或 `V(G) = 判定节点数 + 1`。

**5.5 CMMI**
- 5级：初始级（1）→已管理级（2）→已定义级（3）→定量管理级（4）→优化级（5）。

### 第6章 数据库设计基础知识

**6.1 关系数据库**
- ER图：实体（矩形）、属性（椭圆）、关系（菱形），映射基数1:1/1:N/M:N。
- 关系代数：选择（σ）、投影（π）、连接（⋈）、除（÷）。
- 范式：
  | 范式 | 定义 |
  |:---|:---|
  | 1NF | 属性不可再分 |
  | 2NF | 满足1NF且非主属性完全依赖于候选键 |
  | 3NF | 满足2NF且非主属性不传递依赖于候选键 |
  | BCNF | 所有函数依赖左部包含候选键 |
- 事务ACID：原子性、一致性、隔离性、持久性。
- 并发控制：封锁协议（2PL）、死锁检测/预防、多版本控制（MVCC）。
- 分布式事务：2PC（两阶段提交）、3PC、TCC、Saga、本地消息表。

**6.2 NoSQL与NewSQL**
| 类型 | 代表 | 特点 |
|:---|:---|:---|
| 键值 | Redis | 高性能缓存 |
| 文档 | MongoDB | 灵活Schema |
| 列族 | HBase | 大表、稀疏 |
| 图 | Neo4j | 关系密集型 |
| NewSQL | TiDB | 分布式SQL支持ACID |

**6.3 CAP与BASE**
- CAP：一致性（Consistency）、可用性（Availability）、分区容忍性（Partition Tolerance），三者不可兼得。
- BASE：基本可用（Basically Available）、软状态（Soft State）、最终一致性（Eventually Consistent）。

### 第7章 系统架构设计基础知识（核心章）

**7.1 软件架构风格（必背）**

| 风格 | 核心特征 | 优势 | 不足 | 典型场景 |
|:---|:---|:---|:---|:---|
| **管道-过滤器** | 数据流依次通过各过滤器 | 高复用、易理解 | I/O瓶颈、不适应交互 | Unix Shell、编译器 |
| **数据抽象/面向对象** | 封装、继承、多态 | 易修改、复用 | 间接性增加复杂度 | Java/C++组件 |
| **事件驱动** | 隐式调用、发布-订阅 | 松耦合、高性能 | 调试困难、全局状态难控 | GUI系统、消息队列 |
| **分层** | 每层为上层提供服务 | 关注点分离、可替换 | 层次过多性能损耗 | OSI、Spring MVC |
| **微核** | 核心很小、插件扩展 | 可扩展、可移植 | 插件间通信复杂 | Eclipse、Chrome、OSGi |
| **C2** | 构件+连接件分层消息传递 | 可替换、支持多视图 | 适用范围有限 | 富客户端应用 |
| **解释器** | 虚拟机执行中间语言 | 可移植、动态扩展 | 性能低 | JVM、Lua解释器 |
| **SOA** | 粗粒度服务、ESB | 业务对齐、复用 | 治理复杂、ESB瓶颈 | 企业应用集成 |
| **微服务** | 独立部署、限界上下文 | 技术异构、弹性 | 分布式复杂度 | 互联网平台 |
| **云原生** | 容器、编排、不可变基础设施 | 弹性、自动化 | 技术栈门槛高 | SaaS/PaaS平台 |

**7.2 质量属性与战术**

- **6大质量属性**：
  1. 性能（Performance）：响应时间、吞吐量、延迟
  2. 可用性（Availability）：MTBF、MTTR、容错
  3. 可修改性（Modifiability）：局部化、信息隐藏
  4. 安全性（Security）：认证、授权、审计、保密性
  5. 可测试性（Testability）：可观察、可控制
  6. 易用性（Usability）：用户效率、满意度

- **质量属性战术**：
  - 性能战术：引入并发、维持多个副本、加速资源、限制执行时间、调度策略
  - 可用性战术：冗余、心跳、故障检测与恢复、主动冗余、Passive冗余
  - 可修改性战术：模块化、信息隐藏、接口稳定、运行时绑定
  - 安全性战术：认证、授权、加密、完整性校验、审计日志

**7.3 架构评估方法**

| 方法 | 全称 | 核心理念 | 步骤 |
|:---|:---|:---|:---|
| **SAAM** | Scenario-based Architecture Analysis Method | 基于场景 | 开发场景→描述架构→分类场景→评估→分析 |
| **ATAM** | Architecture Tradeoff Analysis Method | 权衡分析 | 第1步：收集场景 → 第2步：架构驱动 → 第3步：分析 → 第4步：权衡 |
| **CBAM** | Cost Benefit Analysis Method | 成本-效益 | 在ATAM基础上引入经济和成本分析 |

**7.4 质量属性效用树**

- 树形结构：质量属性（根）→ 属性分类（一级）→ 场景（叶子）
- 示例：
  ```
  可用性 ─┬─ 故障检测 ── 场景：心跳间隔<10s
          └─ 故障恢复 ── 场景：故障转移<30s
  性能   ─┬─ 延迟    ── 场景：95%请求<200ms
          └─ 吞吐量  ── 场景：支撑1000并发
  ```

### 第8章 系统质量属性与架构评估

**8.1 质量属性概念**
- 功能性与非功能性：功能性描述"系统做什么"，非功能性描述"系统做得多好"。
- 质量属性场景：**刺激源 → 刺激 → 环境 → 制品 → 响应 → 响应度量**。

**8.2 ATAM详细步骤**
1. 演示阶段：阶段1（场景和需求收集）、阶段2（架构视图描述）
2. 调查阶段：阶段3（属性优先级划分）、阶段4（架构分析）
3. 测试阶段：阶段5（效用树）、阶段6（场景验证）
4. 报告阶段：阶段7（风险/非风险/权衡点/敏感点报告）

**8.3 敏感点、权衡点、风险点**
- 敏感点：某个质量属性对架构某方面敏感（如CPU对性能敏感）。
- 权衡点：影响多个质量属性的决策（如缓存同时影响性能和可用性）。
- 风险点：可能危害达成质量目标的决策。

### 第9章 软件可靠性基础知识

**9.1 基本概念**
- MTBF（平均无故障时间）= MTTF + MTTR
- 可靠性 = MTTF / (MTTF + MTTR)
- 可用性 = 系统正常运行时间 / 总时间

**9.2 可靠性建模**
- 串联系统：R(t) = R1×R2×...×Rn
- 并联系统：R(t) = 1 - (1-R1)×(1-R2)×...×(1-Rn)
- 混联系统：先串后并 / 先并后串

**9.3 容错设计**
- 冗余策略：硬件冗余（双机热备）、软件冗余（N版本）、信息冗余（ECC）、时间冗余（重试）。
- 恢复块、N版本编程、表决机制。

#### 方向五：性能与可靠性设计（对应教材第9章）

**典型题5：系统性能瓶颈分析与优化**

> 某系统单机支撑1000并发时响应时间从50ms飙升至5s，容器CPU使用率99%，数据库连接池打满。请分析瓶颈并给出优化方案。

**答题框架**：
1. 瓶颈定位：CPU耗尽（计算瓶颈）、连接池打满（IO瓶颈）→ 加计算节点 vs 加连接
2. 性能公式：响应时间 = 排队延迟 + 服务时间
3. 优化方案：缓存（减少重复计算）、异步化（MQ削峰）、水平扩展

**参考公式**：
```
- 加速比（Amdahl）：S = 1 / ((1 - F) + F/N)
- QPS = 1000ms / 平均响应时间(ms) × 并发线程数
- 系统吞吐量峰值 = 最小瓶颈环节的吞吐量
```

## 第四部分：论文写作

### 4.1 考试要求

| 项目 | 说明 |
|:---|:---|
| 时长 | **120分钟**（下午第3科，通常15:20-17:20） |
| 题量 | 4道论文题选1道 |
| 字数 | **2000~3000字**（官方要求），含摘要+正文+结论 |
| 形式 | **手写**（多数考区仍为笔试） |
| 满分 | **75分** |
| 合格线 | **45分** |
| 评分维度 | 切题30% + 应用深度20% + 实践性20% + 表达15% + 综合15% |

### 4.2 评分标准详解

| 维度 | 权重 | 评分标准 | 得分关键 |
|:---|:---:|:---|:---|
| **切题** | 30% | 完全覆盖题中所有子论点，不跑题 | 开篇即点题，每段呼应题目要求 |
| **应用深度与水平** | 20% | 技术方案选型理由、决策依据、对比分析 | 不止"用了什么"，还要"为什么选这个" |
| **实践性** | 20% | 有具体项目背景、真实数据、个人角色 | 有时间、规模、QPS、团队人数、上线效果 |
| **表达能力** | 15% | 结构清晰、逻辑顺畅、专业术语准确 | 分节序号清晰，段落≤8行，少废话 |
| **综合与分析** | 15% | 多角度分析、权衡取舍、有反思总结 | 既讲成功也讲不足，体现工程思维 |

**加分项**：紧扣新技术趋势（云原生/服务网格/AI工程化）且有落地案例 → 可+5-10分。

**扣分雷区**：
- 纯理论堆砌，无具体项目 → 直接不及格
- 跑题、答非所问 → 0分
- 摘要部分写无关内容 → 严重扣分
- 字数不足、多余 → 按比例扣分
- 字迹潦草难以辨认 → 最高扣10分

### 4.6 论文实战技巧

**1. 摘要是第一印象**：阅卷老师通常前30秒看摘要+结论+字数。摘要必须：
- 明确项目行业、甲方、规模（**具体！**）
- 明确个人角色
- 点出题目中的所有子论点
- 有量化效果数据

**2. 全文必须有数字**：
```
× 错误："大幅提升了系统性能"
✓ 正确："系统上线后，TPS从800提升至3200，响应时间P99从1.8s降至120ms"
```

**3. 技术选型必须做对比**：
```
× 错误："我们选用了Redis作为缓存"
✓ 正确："考虑到读写比例7:3、数据量约500万key，对比了Redis与Memcached。选Redis的理由：
     1) 支持丰富数据结构（List/Set/ZSet），满足业务复杂计算需求；
     2) RDB+AOF双持久化，避免重启丢数据；
     3) 官方集群方案（Redis Cluster）天然支持分片。Memcached仅支持简单KV且无持久化，不满足场景。"
```

**4. 必须画1-2张图**：系统架构图（必画）、部署图、时序图（选画）。
- 标注关键组件名称、协议、数据流方向
- 手写考试中图要画大、字要正

**5. 结尾必须写反思**：
```
"本方案仍存在不足：①未实现跨AZ自动容灾切换，后续将引入K8s+多云策略；
②监控体系依赖开源Prometheus，告警准确性约85%，计划引入智能告压制。
③存储层单集群上限TB级，未来业务量增长10倍需改为分片集群。"
```

**6. 手写工整**：字不要太小，行距适当。留出20-30分钟收尾检查。

### 4.7 模拟论文题

**题：论微服务架构设计在企业级系统中的应用**

请围绕以下要点展开：
1. 简要叙述你参与的企业级系统项目及您在其中担任的工作。
2. 微服务架构相比于单体架构有哪些优势和挑战？
3. 针对你项目的业务场景，如何进行微服务拆分（限界上下文划分）？
4. 微服务间如何实现通信和数据一致性？
5. 在实施过程中遇到了哪些问题？如何解决的？

**建议练习框架**：

**【摘要】**
2023年3月，本人参与某大型零售企业"全渠道中台"建设项目，担任系统架构师。该项目覆盖线上商城、线下POS、仓储物流三大业务线，日均订单量150万，用户5000万。本文以该项目为例，围绕微服务架构设计展开论述：首先基于DDD的限界上下文将单体应用拆分为12个微服务；其次采用gRPC（同步）+ RocketMQ（异步）的混合通信模式；然后通过Saga模式+TCC两阶段保障分布式事务一致性。系统上线后，日常吞吐从1500 TPS提升至8000 TPS，支撑全年双11峰值2.3万TPS无宕机，证明了微服务架构的有效性。

**【正文】**
一、项目背景与挑战
1.1 项目概述：某零售集团现有系统为2010年建设的单体（Java+Oracle），随着业务快速扩张，出现三大问题：①版本迭代困难（改一行全量部署）；②数据库连接池爆满（Oracle连接池长期超限）；③促销活动期间频繁崩溃（双11部分模块挂了4小时）。
1.2 本人角色：系统架构师，负责微服务拆分方案设计、技术选型、与各业务线技术负责人的方案对齐。
1.3 核心挑战：①如何合理划分服务粒度；②如何在不中断业务的前提下从单体迁移；③分布式环境下的数据一致性如何保证。

二、微服务拆分与架构设计
2.1 限界上下文划分（DDD）
- 战略设计阶段：梳理业务域 → 子域（核心域/支撑域/通用域）
  - 核心域：订单、支付、库存、履约
  - 支撑域：权限、通知、日志
  - 通用域：短信、邮件、文件服务
- 最终划分为12个微服务，每个服务独立数据库（Database per Service）

2.2 通信模式设计
| 通信模式 | 适用场景 | 选型 | 理由 |
|:---|:---|:---|:---|
| 同步（gRPC） | 实时查询、强一致读 | gRPC+Protobuf | 性能是REST的2-3倍，强类型接口 |
| 异步（MQ） | 事件通知、解耦 | RocketMQ | 事务消息、可靠重试、亿级吞吐 |
| 同步（REST） | 开放API | Spring Cloud Gateway | 对外标准、生态成熟 |

2.3 分布式事务方案
- 强一致场景（支付→账户扣款）：TCC模式（Try-Confirm-Cancel）
- 最终一致性场景（下单→库存扣减→通知物流）：Saga+本地消息表
- 关键：引入Seata框架进行全局事务协调，结合Retry+补偿机制

三、实施效果与反思
3.1 效果数据（上线6个月后对比）：
| 指标 | 迁移前（单体） | 迁移后（微服务） |
|:---|:---:|:---:|
| 日均TPS | 1,500 | 8,000 |
| 双11峰值TPS | 3,000（近崩溃） | 23,000（正常） |
| 发布周期 | 2周/次 | 每天10+次 |
| 单次发布影响范围 | 全系统 | 1~2个服务 |

3.2 反思：
- 拆分粒度偏细（12个服务初期增加了运维成本），建议对低变更率服务合并。
- 分布式追踪用SkyWalking初步覆盖，但全链路压测时发现Trace采样率50%影响性能，后续调整为自适应采样。
- Saga模式在极端情况（网络分区）仍有数据不一致窗口，计划引入定期的对账任务兜底。

四、结论
本文介绍了微服务架构在某零售企业全渠道中台中的实践。通过DDD限界上下文拆分、gRPC+MQ混合通信和Seata分布式事务，解决了单体的扩展性和可靠性问题，验证了微服务架构在企业级系统中的可行性。后续将继续优化服务治理和服务网格（Istio）的引入。
